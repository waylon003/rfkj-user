# rfkj-user 接入判断台账

## 结论

当前项目**方向上**和 `redeem-wxapp` 一致：页面先固定为前端模型，接口只在 `service / mock / adapter` 层对接。

但它**还不是** `redeem-wxapp` 那种已经大规模真实接入、并被大量适配规则验证过的成熟形态。更准确地说：

- 大多数页面已经不会直接依赖后端原始字段
- 真实接口变化，优先改 `services/*` 的映射层
- 只有当接口行为和页面模型冲突时，才回头改页面模型

## 当前项目的真实状态

### 已经成立的部分

- 页面统一消费 `services/*` 导出的页面模型
- mock 统一经由 `src/mock` 和 `src/utils/mock.ts` 收口
- 登录链路已经有真实接口入口，且在 `services/auth/api.ts` 内做了字段映射

### 还没完全成立的部分

- 绝大多数业务模块仍然是 mock
- `auth` 是当前唯一明确走真实接口的模块
- 门店、优惠券、部分会员辅助页仍以本地常量或静态模型为主，后续接真时更可能改页面模型

## 与 `redeem-wxapp` 的差异

`redeem-wxapp` 已经形成了明确的执行规则：

- 页面不直接消费后端原始结构
- 必须经过 `service / adapter` 映射
- 行为不一致时，页面继续保留 mock

`rfkj-user` 现在更像是“规则先行”，还没有进入 `redeem-wxapp` 那种“规则 + 大量真实接口落地”的阶段。

| 排行页 | `services/ranking` + mock | 是 | 低 |

## 具体判断

### 基本可以认为“只改 service / mock”的页面

- `src/pages/home/index.vue`
- `src/pages/profile/index.vue`
- `src/pages/profile/info.vue`
- `src/pages/profile/gift-records.vue`
- `src/pages/profile/gifts.vue`
- `src/pages/profile/bill.vue`
- `src/pages/activity/index.vue`
- `src/pages/activity/detail.vue`
- `src/pages/ranking/index.vue`
- `src/pages/member/gift.vue`
- `src/pages/coin/*`
- `src/pages/points/exchange.vue`

这些页面已经是“前端模型驱动”，接口字段进来后，优先在 `services/*` 做适配。

### 接真时更容易改页面模型的页面

- `src/pages/home/store-select.vue`
- `src/pages/home/store-detail.vue`
- `src/pages/coupon/center.vue`
- `src/pages/member/tickets.vue`

这些页面目前更偏“演示型 / 占位型”，字段结构还没真正收口成稳定契约。

### 只需要保留静态内容的页面

- `src/pages/auth/member-terms.vue`
- `src/pages/auth/privacy-policy.vue`

## 未来接真时的执行规则

1. 先看页面是否已经有稳定的 `Data` 类型
2. 能映射就只改 `services/*`
3. 字段缺失，只允许在适配层补空值或默认值
4. 接口行为和页面逻辑冲突时，优先保留页面模型，不强行伪造后端能力
5. 只有确定页面模型不成立，才改页面

## 结论

`rfkj-user` 不是“纯 mock 页面拼装”，而是已经在往“页面模型固定、接口只做适配”的方向走。

所以答案是：

- **是的，方向上很像 `redeem-wxapp`**
- **但现在还不能说后续接口怎么变都完全不影响页面模型**
- **更准确的说法是：大多数页面未来应尽量只改 mock / service 层，少数模型不稳定页面仍可能要改**

## 2026-05-13 最新对齐补充

### 可以直接接入

- `home`：`GET /wechat/member/banners`
- `auth`：`onlogin`、`GetUserPhoneNumber`、`userinfo`、`updateaddress`
- `profile/info.vue`：资料展示与资料保存

### 可以部分接入，但建议先保留 mock 外壳

- `points/exchange.vue`：分类、商品列表、积分余额
- `profile/index.vue`：顶部会员身份区
- `profile/gifts.vue`：待提货 / 已提货列表
- `profile/gift-records.vue`：赠送 / 收到的基础流水
- `coin/recharge.vue`：余额、套餐列表
- `member/gift.vue`：赠送提交动作

### 继续保持 mock

- `profile/bill.vue`
- `ranking/index.vue`
- `activity/index.vue`
- `activity/detail.vue`
- `coupon/*`
- `member/tickets.vue`
- `home/store-select.vue`
- `home/store-detail.vue`

### 原因归类

- 缺少字段：
  - `points` 的兑换提交隐藏字段
  - `profile` 的领取状态、二维码、账单聚合字段
  - `purchase` 的特惠、抵扣、支付面板字段
- 缺少接口：
  - 活动域接口
  - 机台 / 多维排行榜接口
  - 统一账单接口
  - 优惠券接口
  - 购币支付闭环接口
  - 用户获取已注册门店列表接口

### 门店模型最新口径

- 门店选择页不再展示全门店列表，而是只展示“用户已注册的门店”
- 首次登录若尚未注册任何门店会员，应展示空状态，并引导用户扫码注册门店会员
- 列表排序规则为：按定位距离最近优先
- 门店卡片需要展示“距我 x 米 / km”字段
- 当前后端尚未提供“用户已注册门店列表”接口，因此 `store-select.vue` / `store-detail.vue` 仍保持 mock facade，不做真实接入
- 首页 Banner 已调整为游客态也可直接请求，且支持可选 `storeId` 参数
